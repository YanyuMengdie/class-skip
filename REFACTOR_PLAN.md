# class-skip 屎山重构计划

> 这份文档是 Criss 和 Claude 一起制定的 class-skip 项目重构方案。
> 重构期间随时可以打开这份文档，查看目标、流程、决策原则。

---

## 一、为什么要重构（目的）

**核心目的**：让改 bug 和加新功能的速度变快、风险变小。

现在屎山的代价：

- 想改一个小东西，发现牵一发动全身，改完还崩了别的
- 想加新功能，发现要塞进哪都不合适
- 一个组件 1500 行，找一段逻辑要滚屏 5 分钟
- 让 AI（Claude Code）帮忙改也力不从心，因为代码没结构 AI 也理不清

**重构本身不创造新功能**。重构是为了之后创造功能时跑得飞快。

---

## 二、重构目标（具体衡量标准）

| 目标 | 现在 | 之后 |
|------|------|------|
| 单个组件最大行数 | 可能 1500+ | < 400 行 |
| 找一个功能要打开几个文件 | 可能 5-10 个 | 1-3 个 |
| 加新模块要改多少个旧文件 | 可能 10+ | 1-2 个 |
| TypeScript 严格模式 | 可能没开 | 开启 |
| 死代码占比 | 估计 10-30% | < 5% |
| 让 Claude Code 加新功能要花的时间 | 长，且容易乱改 | 短，定向修改 |

---

## 三、重构后的样子（结果预期）

打开项目第一感觉应该是：

- 文件夹结构一目了然（`features/socratic/`、`features/feynman/`、`features/exam/` 等）
- 想改什么功能，3 秒内能找到对应文件
- 加新模块时有清晰模板可以参考
- 让 Claude Code 改东西，修改范围明显变小、更精准
- 跑 `npx tsc --noEmit`，0 错误

**功能完全不变**。用户看到的界面、能做的操作、数据……和现在一模一样。变的只是底下的代码组织方式。

> 这点非常重要——如果重构改变了用户能感知的功能，那是 feature work，不是重构。

---

## 四、时间估计

诚实说：取决于屎的程度。审计跑完才能精确估计。

基于对 class-skip 的了解，**保守估计**：

| 阶段 | 需要的时间 |
|------|-----------|
| 阶段 1：审计 + 看报告决策 | 1 小时 |
| 阶段 2：清理死代码 | 2-3 小时（分 2-3 次完成） |
| 阶段 3：重组目录结构 | 3-4 小时（分 2-3 次完成） |
| 阶段 4：拆分大组件 + 抽业务逻辑 | 8-15 小时（分 5-10 次完成） |
| 阶段 5：类型严格化 | 3-5 小时 |

**总计：17-28 小时**，分散在 2-4 周完成。每天 1-2 小时就够。

不需要一次做完。每次结束都是稳定状态。

## 五、五个阶段路线图

```
阶段 1：审计（不改代码）
   └─ Claude Code 全面分析项目，给出"诊断报告"
   └─ 看完知道屎在哪、有多少、怎么排序处理

阶段 2：清理死代码和重复
   └─ 删掉没人用的文件、合并重复逻辑
   └─ 项目立刻变小变干净，后面所有重构都更轻松

阶段 3：重组目录结构
   └─ 按功能把文件归类（比如所有考试相关的放一起）
   └─ 找文件再也不用满项目搜索

阶段 4：拆分巨型组件 + 业务逻辑剥离
   └─ 把 1000+ 行的组件拆成小块
   └─ 把"算法"和"界面"分开（屎山的核心病根）

阶段 5：补类型和测试
   └─ TypeScript 严格化 + 给核心算法加保护网
   └─ 之后改代码不用怕改崩
```

## 六、典型一次重构是什么样的（情景示例）

```
Criss：今天想搞 SkimPanel.tsx，它太大了

Claude：好，这个组件目前 800 多行，建议拆成 4 部分：
        - SkimPanel 主框架（保持）
        - SkimContentView 内容显示
        - SkimQuestionList 问题列表
        - useSkimState 状态逻辑 hook

        问几个问题：
        Q1: 当前页面定位（pageMark）的逻辑要不要单独抽出来？
        Q2: 有些 ChatHug 和 SkimPanel 耦合太深，要不要顺便处理？
        Q3: 之前说想给 Skim 加"重读模式"，这次要不要预留接口？

Criss：Q1 抽出来；Q2 这次先不动；Q3 暂时不预留

Claude：明白。让 Claude Code 按这个方案改，改完之后：
        1. 跑 tsc 检查（保证 0 错误）
        2. Criss 测以下 3 个功能：
           - 进入 Skim 页面
           - 滚动时页面定位
           - 点击问题跳转

        通过则 commit 消息：
        "refactor(skim): 拆分 SkimPanel 为 4 个文件"

[Claude Code 干活，Criss 喝水]

Claude：改好了，测一下那 3 个功能

Criss：滚动定位失效了

Claude：找到了，useSkimState 里 useEffect 依赖项没传对，现在改回来

[修复]

Criss：现在好了

Claude：commit 并 push。这一小步完成。今天还继续吗？

Criss：累了，明天再说

Claude：好，今天的成果已经 push 到 GitHub
        main 分支干净，refactor 分支多了一个 commit
        随时可以接着干
```

**关键观察**：每一步都问 Criss、每一步都能停、每一步都安全。

## 七、Criss 需要做的决策

### 审计阶段
- 哪些文件确认是死代码可以删？
- 哪些"看起来重复"的组件其实有不同用途，不能合并？
- 重构优先级先做哪个模块？

### 每次具体改动前
- 这个组件应该拆成几块？
- 这个功能的"职责边界"在哪里？（比如 SkimPanel 该不该管 PDF 渲染）
- 起个新文件夹叫什么名字？
- 哪些旧代码彻底删，哪些注释保留以备查阅？

### 功能边界上
- 这次重构顺手把这个 bug 修了？还是只动结构不动行为？
- 这个组件之前有些"看起来奇怪"的逻辑，是 bug 还是故意的？
- 这两个看起来一样的函数，要合并还是保留？

**Claude 永远不替 Criss 决定产品逻辑**。Claude 只决定纯技术的事（TypeScript 怎么写、文件怎么组织）。涉及"这个功能应该怎么用"的事都问 Criss。

## 九、四道安全网

### 第 1 道：GitHub 永远的"还原点"
GitHub 上的版本是已知能跑的、有所有功能的。
万一改崩了，最坏情况一条命令回到这一刻：
```bash
git reset --hard origin/main
```

### 第 2 道：每改一小步就 commit
每改完一个小东西就：
1. 跑 `npm run dev` 看页面还能不能用
2. 能用 → `git commit` 存档
3. 不能用 → `git checkout .` 撤销，重来

### 第 3 道：在 refactor 分支上做，main 分支不动
所有重构在新分支 `refactor` 上做，主分支 `main` 始终保持现在能跑的状态。
新分支哪怕拆崩了，随时可以切回 main 分支，原功能一点没受影响。

### 第 4 道：先审计、不动手
阶段 1 只读代码、写报告，不改任何东西。
看完报告再决定要不要继续、从哪开始。

## 十、测试策略（不靠手动全测）

### 第 1 层：TypeScript 类型检查（机器干，1 秒）
每次改完跑 `npx tsc --noEmit`。
能抓住 80% 的低级崩溃——比如改了函数参数名忘了改调用处。

### 第 2 层：定向测试（Criss 干，2-5 分钟）
每次重构前 Claude 会明确告诉 Criss：
> "这次改动影响的范围是 X、Y、Z 三个功能，请测一下这三个就行。"

不需要每次全测，只测受影响的范围。

### 第 3 层：完整冒烟测试（Criss 干，15-20 分钟）
不是每次 commit 都做，而是**每完成一个阶段后做一次**。
照着"必测功能清单"过一遍，全过 → 这阶段成功。

## 十一、几个核心原则

### 1. "小步快跑"原则
永远不要一次改一大堆然后才 commit。每改完一个独立的小改动就 commit。
万一改崩了，git 能精确告诉是哪次改动导致的。

### 2. "改完测一下"原则
每次改完，跑 `npm run dev`，浏览器看看页面还能不能用。
能用 → commit；不能用 → 立刻回滚不要硬改。

### 3. "AI 不是神"原则
Claude Code 会犯错，会自作主张改不想改的东西。所以：
- 每次它说"我改了 XX"，看一眼它具体改了什么（VS Code 会高亮）
- 不确定的就问它"为什么这样改"
- 改之前先 commit 当前状态，万一搞砸了能回滚

### 4. "不追求完美"原则
目标是**让代码可维护**，不是**让代码完美**。
能跑、能改、能加新功能 = 成功。
class-skip 的核心价值是**功能本身和迭代速度**，不是代码美学。
